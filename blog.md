# 지출결의서 보기 페이지 로딩 속도 개선 트러블슈팅

## 1. 문제 상황

지출결의서 보기 페이지는 회계팀과 마스터 계정이 가장 자주 사용하는 화면 중 하나다.  
목록을 여러 페이지 넘겨가며 확인하고, 각 문서의 상세 내용을 반복해서 열어보는 사용 패턴이 많다.

초기에는 문서 수가 많지 않아 큰 문제가 없었지만, 데이터가 쌓이기 시작하면서 다음 문제가 한꺼번에 드러났다.

1. 회계팀 계정에서 작성자명이 `작성자 미상`으로 보이는 경우가 있었다.
2. 목록을 볼 때마다 API를 다시 호출하면서 체감상 5~6초 정도 기다리는 느낌이 컸다.
3. 페이지를 넘길 때도 다시 로딩하는 것처럼 보여 흐름이 끊겼다.
4. 상세 보기를 눌렀을 때 로딩 중이라는 피드백이 거의 없어 멈춘 것처럼 느껴졌다.
5. 이미 봤던 페이지나 이미 열었던 상세 문서를 다시 봐도 빨라지는 느낌이 거의 없었다.

즉, 단순히 "응답 시간을 줄이는 것"만으로는 부족했고,  
사용자가 실제로 느끼는 체감 속도를 함께 개선해야 하는 상황이었다.

## 2. 이번 트러블슈팅의 목표

이번 작업에서 목표로 잡은 기준은 명확했다.

- 첫 목록 진입이 너무 무겁지 않아야 한다.
- 페이지 재방문은 가능한 한 즉시 보여야 한다.
- 상세 첫 열기에는 명확한 로딩 피드백이 있어야 한다.
- 상세 재열기는 캐시를 활용해 빠르게 열려야 한다.
- 권한 구조는 그대로 유지하면서 작성자명 표시 안정성을 확보해야 한다.

핵심은 "모든 경우를 무조건 빠르게"가 아니라,  
"처음은 정확하게, 다시 볼 때는 빠르게"라는 방향으로 구조를 바꾸는 것이었다.

## 3. 기존 구조의 한계

개선 전 구조에서는 목록과 상세를 최적화하는 책임이 충분히 분리되어 있지 않았다.

- 목록 화면과 상세 화면이 사실상 한 덩어리처럼 느껴졌다.
- 작성자명처럼 권한 영향을 받는 데이터가 프론트에서 불안정하게 보일 여지가 있었다.
- 목록 조회와 상세 조회가 모두 느릴 때 어느 쪽이 병목인지 구분하기 어려웠다.
- 서버 페이지네이션이 없어서 첫 조회 비용을 계속 줄이기 어려웠다.
- 캐시가 없어 같은 데이터를 다시 볼 때도 매번 네트워크를 기다려야 했다.

정리하면, "한 번 불러온 데이터를 다시 잘 쓰는 구조"가 없었고,  
"지금 로딩 중인지" 보여주는 UX도 부족했다.

## 4. 해결 전략

이번에는 단순한 마이크로 최적화보다 구조 정리를 먼저 했다.

1. 권한과 데이터 조합 책임은 서버에 둔다.
2. 목록과 상세를 API 단에서 분리한다.
3. 목록과 상세 각각에 맞는 캐시 전략을 따로 적용한다.
4. 캐시가 있어도 사용자가 그 이점을 체감할 수 있게 UX를 같이 손본다.
5. 마지막에는 실제로 빨라졌는지 숫자로 확인할 수 있도록 측정 도구를 넣는다.

또 하나의 원칙은, 아직 확실한 효과가 보장되지 않은 `PC hover prefetch`보다는  
PC와 모바일 모두에서 공통으로 체감되는 개선부터 먼저 적용하는 것이었다.

## 5. 실제 적용한 개선

### 5-1. 작성자명 표시 문제를 서버 책임으로 정리

회계팀 계정에서 작성자명이 `작성자 미상`으로 보이던 문제는  
프론트에서 조합하던 데이터를 서버 응답 기준으로 정리하면서 해결했다.

- 권한 구조는 유지했다.
- 회계팀 계정에서도 작성자명이 안정적으로 보이게 했다.
- 목록과 상세 모두 같은 기준으로 작성자명을 보여줄 수 있게 됐다.

이 단계는 속도 개선이라기보다, 이후 최적화를 안전하게 진행하기 위한 기초 공사에 가까웠다.

### 5-2. 목록 API와 상세 API 분리

목록용 API와 상세용 API를 분리했다.

- 목록 API는 리스트 화면에 필요한 경량 정보만 내려준다.
- 상세 API는 본문, 항목, 영수증 등 상세 모달에 필요한 데이터만 내려준다.

이렇게 나누면서 두 가지가 좋아졌다.

- 목록은 훨씬 가볍게 다룰 수 있게 됐다.
- 상세는 별도 캐시와 별도 로딩 UX를 적용하기 쉬워졌다.

### 5-3. 서버 기준 페이지네이션 도입

PC와 모바일 모두 한 번에 5건씩만 보이도록 서버 기준 페이지네이션을 적용했다.

- 한 번에 모든 문서를 내려받지 않도록 변경
- 데이터가 늘어나도 목록 조회 부담을 일정하게 유지
- PC와 모바일에 동일한 페이지 기준 적용

이 단계만으로도 초기 목록 조회 부담은 눈에 띄게 줄었다.

### 5-4. 목록 캐시 적용

목록에는 `사용자 + 페이지 번호 + 필터값` 기준 메모리 캐시를 적용했다.

- 최근에 본 페이지는 다시 열 때 즉시 보여준다.
- 캐시가 fresh하면 네트워크 호출 없이 바로 재사용한다.
- 캐시가 stale이면 일단 화면을 먼저 보여주고 뒤에서만 최신 데이터를 다시 확인한다.

즉, 목록은 `stale-while-revalidate` 방식으로 바꿨다.

### 5-5. 페이지 이동 UX 개선

캐시가 있어도 화면이 다시 크게 로딩되는 것처럼 보이면 체감 성능은 좋아지지 않는다.  
그래서 페이지 이동 UX를 캐시 전략과 함께 조정했다.

- 캐시된 페이지가 있으면 버튼 클릭 직후 즉시 목록 교체
- 전체 오버레이 로딩 대신 필요한 경우에만 `최신화 중` 표시 노출
- 다음 페이지는 백그라운드에서 미리 불러오도록 구성

이 변경으로 `1페이지 -> 2페이지 -> 다시 1페이지` 같은 흐름이 훨씬 자연스러워졌다.

### 5-6. 상세 보기 로딩 피드백 개선

상세 보기에서는 "지금 로딩 중"이라는 사실이 바로 보여야 했다.

- 상세 보기 버튼을 누르면 모달이 먼저 열린다.
- 데이터가 아직 없으면 모달 안에 스켈레톤과 로딩 안내를 보여준다.
- 클릭한 행의 상세 보기 버튼에는 작은 스피너를 표시한다.
- 늦게 도착한 응답이 현재 화면을 덮어쓰지 않도록 요청 경쟁도 방어했다.

이 단계 이후부터는 사용자가 "멈췄다"고 느끼는 경우가 크게 줄었다.

### 5-7. 상세 캐시 적용

상세 데이터는 `사용자 + reportId` 기준으로 메모리 캐시를 적용했다.

- 한 번 연 상세는 다시 열 때 거의 즉시 보여준다.
- fresh 캐시라면 네트워크 호출 없이 바로 연다.
- stale 캐시라면 cached 상세를 먼저 보여주고, 뒤에서 최신 정보를 다시 확인한다.
- 상세 모달 헤더에는 필요한 경우 `최신 정보 확인 중` 표시를 노출한다.

목록과 상세 모두 같은 원칙으로 맞춘 셈이다.

- 처음은 정확하게
- 재방문은 빠르게

### 5-8. 캐시 무효화 범위 확장

캐시를 넣고 나면 더 중요한 것은 "언제 비워야 하는가"다.

이번에는 다음 흐름까지 캐시 무효화 범위를 확장했다.

- 삭제 후 목록/상세 캐시 무효화
- 체크 상태 변경 후 관련 캐시 무효화
- 작성, 수정, 제출 후 `/expense` 복귀 시 세션 신호를 읽어 관련 캐시 무효화
- 사용자 전환 시 이전 사용자 캐시 정리

즉, "빠르지만 오래된 데이터"가 보이는 문제를 줄이기 위해  
캐시 자체보다 캐시 정합성 규칙을 먼저 신경 썼다.

### 5-9. 성능 측정 도구 추가

체감상 빨라진 느낌만으로는 트러블슈팅 기록이 충분하지 않다고 판단해서  
성능 측정 도구도 함께 넣었다.

현재는 `?expensePerf=1` 쿼리로 측정을 켤 수 있고, 다음 지표를 기록한다.

- `list_display`
- `list_refresh`
- `detail_display`
- `detail_refresh`

또한 `scenario`와 `cacheState`를 함께 기록해서 다음을 구분할 수 있게 했다.

- 첫 진입인지 재방문인지
- cache miss인지 fresh hit인지 stale hit인지

콘솔에서는 다음처럼 요약을 볼 수 있다.

```js
window.__expenseReportPerf.summary()
window.__expenseReportPerf.read()
window.__expenseReportPerf.clear()
```

## 6. 로컬 측정 결과

로컬 개발 환경에서 실제로 찍힌 측정 로그를 기준으로 보면,  
이제 성능은 "모든 경우가 똑같이 느린 구조"가 아니라 "cold path와 warm path가 분리된 구조"에 가까워졌다.

이번 로컬 측정에서 확인한 `list_display` 예시는 다음과 같았다.

| 시나리오 | 예시 측정값 | 해석 |
| --- | ---: | --- |
| cold path | `6430.3ms` | 캐시가 없고 네트워크/API 영향을 직접 받는 첫 조회 |
| cold path | `2487.8ms` | 첫 조회지만 이전보다 부담이 줄어든 구간 |
| semi-warm path | `357.1ms` | 이미 일부 상태가 준비된 흐름에서 빠르게 표시된 구간 |
| warm path | `0ms` | 캐시 hit로 즉시 표시 |
| warm path | `0.1ms` | 캐시 hit로 사실상 즉시 표시 |

이 수치를 보면 중요한 포인트가 드러난다.

1. 첫 조회는 여전히 서버 상태와 네트워크 영향을 받는다.
2. 하지만 같은 페이지를 다시 볼 때는 거의 즉시 표시된다.
3. 즉, 기존처럼 "매번 5~6초를 기다리는 경험"에서 벗어났다는 것이 핵심이다.

이번 개선의 성과는 "모든 요청을 무조건 1초 이하로 만들었다"가 아니다.  
대신 "처음 한 번의 비용과, 다시 볼 때의 비용을 분리했다"는 데 있다.

그 결과 사용자 입장에서는 실제 업무에서 더 자주 일어나는 재방문 흐름이 훨씬 빨라진다.

## 7. 이번 작업에서 얻은 정리

이번 트러블슈팅을 통해 다시 확인한 점은 다음과 같다.

### 7-1. 응답 시간과 체감 속도는 다르다

첫 요청이 여전히 느릴 수 있어도,  
재방문을 캐시로 즉시 보여주면 사용자는 전체 시스템이 훨씬 빨라졌다고 느낀다.

### 7-2. 캐시는 넣는 것보다 무효화 규칙이 더 어렵다

목록 캐시와 상세 캐시를 넣는 것 자체는 어렵지 않았다.  
더 중요한 것은 작성, 수정, 제출, 삭제, 체크 변경 이후 어떤 캐시를 비울지 정하는 일이었다.

### 7-3. 권한 구조는 서버에 두는 편이 안전하다

작성자명처럼 권한과 연결된 데이터는  
프론트에서 조합하려고 할수록 예외가 늘어나고 유지보수가 어려워진다.

### 7-4. 측정 도구가 있어야 개선이 기록으로 남는다

이번에는 `expensePerf` 측정 도구를 추가하면서  
"느낌상 빨라졌다"를 넘어서 "어떤 흐름이 얼마나 좋아졌는지"를 비교할 수 있게 됐다.

## 8. 남은 과제

현재 시점에서 남은 핵심 과제는 다음 정도로 정리할 수 있다.

### 8-1. 코드 리팩토링

지금 `ExpenseReport.jsx`는 역할이 많이 모여 있다.

- 목록 조회
- 페이지네이션
- 목록 캐시
- 상세 캐시
- 로딩 상태 제어
- 체크 상태 변경
- 삭제
- 성능 측정

기능은 동작하지만, 유지보수성과 테스트 편의성을 생각하면 다음 단계에서는 분리할 가치가 크다.

- `useExpenseReportList`
- `useExpenseReportDetail`
- 캐시 유틸 분리
- 성능 측정 유틸과 페이지 로직의 연결 단순화

### 8-2. 테스트 보강

캐시와 요청 경쟁 방어가 들어간 만큼 수동 확인만으로는 한계가 있다.

- 캐시 키 생성 테스트
- TTL 판정 테스트
- 캐시 무효화 테스트
- 재방문 즉시 표시 흐름 테스트
- 늦게 도착한 응답 무시 테스트

### 8-3. 실서비스 기준 측정 데이터 축적

로컬 측정은 개선 방향을 검증하는 데 유효했지만,  
최종적으로는 배포 환경에서의 반복 측정 데이터도 쌓아야 한다.

- 실제 네트워크 환경 기준
- 실제 사용자 권한 기준
- 실제 문서 개수 기준

이 데이터가 쌓이면 TTL이나 prefetch 범위도 더 정교하게 조정할 수 있다.

## 9. 결론

이번 작업은 단순한 "속도 개선"이라기보다  
"목록과 상세를 따로 최적화할 수 있는 구조를 만들고, 사용자가 실제로 느끼는 속도를 개선한 작업"에 가까웠다.

정리하면 이번에 확보한 것은 다음과 같다.

- 권한 구조를 유지한 작성자명 표시 안정성
- 목록/상세 API 분리
- 서버 페이지네이션 기반 목록 경량화
- 목록 캐시와 페이지 이동 UX 개선
