# 지출결의서 보기 페이지 로딩 속도 개선 트러블슈팅

## 1. 문제 정의

지출결의서 보기 페이지는 회계팀과 마스터 계정이 가장 자주 확인하는 화면 중 하나다.  
목록을 여러 페이지 넘겨가며 확인하고, 각 문서의 상세 내용을 반복해서 열어보는 사용 패턴이 많다.

이번 개선 작업을 시작한 계기는 크게 세 가지였다.

1. 작성자명이 권한에 따라 정상적으로 보이지 않는 문제가 있었다.
2. 지출결의서 수가 늘어나면서 목록 조회 속도와 화면 체감 밀도가 함께 불편해지기 시작했다.
3. 상세 보기를 눌렀을 때 로딩 중인지 알기 어려워, 사용자가 화면이 멈춘 것으로 오해할 수 있었다.

핵심은 단순히 API 응답 시간을 줄이는 것이 아니었다.  
사용자가 실제로 느끼는 속도, 즉 체감 성능을 함께 개선하는 것이 목표였다.

- 첫 진입 시 목록이 너무 무겁지 않아야 한다.
- 페이지 이동 시 기다리는 느낌이 최소화되어야 한다.
- 상세 보기를 눌렀을 때 즉시 반응이 보여야 한다.
- 이미 본 페이지와 이미 연 상세 문서는 최대한 빠르게 다시 보여야 한다.

## 2. 개선 전 구조와 한계

초기 구조에서는 목록 최적화와 상세 최적화를 충분히 분리해서 가져가기 어려웠다.

- 목록 화면과 상세 화면이 각자 다른 방식으로 최적화되기 어려웠다.
- 권한 구조와 작성자명 조합 로직이 프론트에서 불안정하게 보일 여지가 있었다.
- 목록이 느린지, 상세가 느린지, 혹은 둘 다 느린지 병목을 구분하기가 어려웠다.
- 이미 본 데이터를 다시 확인할 때도 캐시 이점이 거의 없었다.

즉, 구조를 정리하지 않은 상태에서 단순 로딩 스피너만 추가하는 것으로는 충분하지 않았다.

## 3. 적용한 개선 사항

### 3-1. 서버 기준 페이지네이션 도입

PC와 모바일 모두 한 번에 5건씩만 보이도록 서버 기준 페이지네이션을 적용했다.

- 한 번에 모든 문서를 내려받지 않도록 바꿔 초기 응답 부담을 줄였다.
- 문서 수가 늘어나도 목록 조회 비용이 일정 범위 안에서 유지되도록 만들었다.
- PC와 모바일이 같은 페이지 단위를 기준으로 동작하도록 정리했다.

### 3-2. 목록 API와 상세 API 분리

목록용 API와 상세용 API를 분리했다.

- 목록 API는 리스트 화면에 필요한 경량 정보만 반환한다.
- 상세 API는 본문, 항목, 영수증 등 상세 모달에 필요한 데이터만 반환한다.

이렇게 나누면 목록은 더 가볍게, 상세는 더 정확하게 최적화할 수 있다.

### 3-3. 작성자명 조회를 서버에서 안전하게 조합

회계팀 계정에서 작성자명이 `작성자 미상`으로 보이던 문제를 해결하기 위해, 작성자명 조합 책임을 서버 쪽 응답으로 정리했다.

- 권한 구조를 유지하면서도 회계팀 계정에서 작성자명이 안정적으로 보이게 했다.
- 프론트는 복잡한 예외 처리보다 표시 로직에 집중할 수 있게 됐다.
- 목록과 상세 모두 같은 기준으로 작성자명을 보여줄 수 있게 됐다.

### 3-4. 목록 캐시 적용

목록은 `사용자 + 페이지 번호 + 필터값` 기준으로 메모리 캐시를 두도록 변경했다.

- 최근에 본 페이지는 다시 열 때 즉시 보여준다.
- 캐시가 신선한 동안에는 네트워크 호출 없이 바로 재사용한다.
- 캐시가 오래된 경우에도 일단 즉시 보여준 뒤, 뒤에서만 최신 데이터를 다시 확인한다.

이 구조는 `stale-while-revalidate`에 가깝다.  
즉, “먼저 보여주고, 필요하면 조용히 새로고침”하는 방식이다.

### 3-5. 페이지 이동 UX 개선

목록 캐시가 있어도 사용자 눈에 다시 로딩되는 것처럼 보이면 개선 효과가 반감된다.  
그래서 페이지 이동 UX도 함께 조정했다.

- 캐시된 페이지가 있으면 버튼 클릭 직후 즉시 목록을 교체한다.
- 전체 화면을 다시 막는 방식 대신, 필요한 경우에만 작은 `최신화 중` 표시를 노출한다.
- 다음 페이지는 백그라운드에서 미리 불러오도록 구성해 재방문 속도를 높였다.

이 변경으로 `1페이지 -> 2페이지 -> 다시 1페이지`처럼 오가는 흐름이 훨씬 자연스러워졌다.

### 3-6. 상세 보기 로딩 피드백 개선

상세 보기에서는 “로딩이 걸리고 있다”는 사실을 바로 느낄 수 있도록 UX를 바꿨다.

- 상세 보기 버튼을 누르면 모달이 먼저 열린다.
- 데이터가 아직 없으면 모달 안에 스켈레톤과 로딩 안내를 보여준다.
- 클릭한 행의 상세 보기 버튼에는 작은 스피너를 표시한다.
- 중복 클릭이나 늦게 도착한 응답이 화면을 덮어쓰지 않도록 요청 경쟁도 방어했다.

이제 사용자는 “멈춘 것 같다”는 느낌 대신 “불러오는 중이구나”를 바로 인지할 수 있다.

### 3-7. 상세 캐시 적용

상세 데이터는 `사용자 + reportId` 기준으로 메모리 캐시를 두도록 변경했다.

- 한 번 연 상세 문서는 다시 열 때 거의 즉시 보여준다.
- 캐시가 신선한 동안에는 네트워크 호출 없이 바로 연다.
- 캐시가 오래된 경우에도 일단 cached 상세를 보여주고, 뒤에서 최신 정보를 다시 확인한다.
- 상세 모달 헤더에는 필요한 경우 `최신 정보 확인 중` 표시를 보여준다.

목록과 상세 모두 “처음은 정확하게, 재방문은 빠르게”라는 같은 원칙으로 정리한 셈이다.

### 3-8. 캐시 무효화 보강

캐시를 도입하면 최신성과 정합성을 함께 관리해야 한다.  
그래서 삭제와 체크 상태 변경 이후에는 관련 캐시를 비우도록 보강했다.

- 목록 캐시는 현재 사용자 기준으로 무효화한다.
- 상세 캐시는 해당 문서 기준으로 무효화한다.
- 사용자 전환 시 이전 사용자 캐시도 정리해 세션 간 데이터가 섞이지 않도록 했다.

## 4. 왜 이 방법을 선택했는가

이번 개선은 “응답 시간이 몇 ms 줄었는가”보다 “사용자가 얼마나 덜 기다리게 느끼는가”를 우선순위로 잡았다.

- 첫 진입만 빠른 것보다 페이지 이동과 상세 재열림이 부드러운 것이 실제 업무 UX에 더 중요했다.
- 서버 페이지네이션만으로는 네트워크 왕복 자체가 사라지지 않기 때문에 캐시가 함께 필요했다.
- 작성자명과 권한 구조 같은 민감한 기준은 서버에서 정리하고, 프론트는 체감 속도 개선에 집중하는 편이 안전했다.
- 아직 실제 사용 여부가 불확실한 `PC hover prefetch`보다, 모든 기기에서 공통으로 체감되는 개선을 먼저 적용하는 편이 맞았다.

## 5. 현재까지의 결과

현재까지 확인된 변화는 다음과 같다.

- 회계팀 계정에서도 작성자명이 정상적으로 표시된다.
- 목록 조회는 서버 페이지네이션과 목록 API 경량화 덕분에 초기 부담이 줄었다.
- 이미 본 목록 페이지는 다시 열 때 훨씬 빠르게 보인다.
- 이미 연 상세 문서는 다시 열 때 즉시 표시되는 수준으로 체감 속도가 좋아졌다.
- 상세를 처음 열 때도 로딩 상태가 명확하게 보여 사용자 혼란이 줄었다.
- 페이지 이동과 상세 전환에서 요청 경쟁으로 인한 덮어쓰기 가능성이 줄었다.

즉, 이번 작업은 단순한 “속도 개선”을 넘어,  
목록과 상세를 각각 따로 다룰 수 있는 구조를 만들고 체감 성능까지 끌어올린 과정이라고 볼 수 있다.

## 6. 남은 작업 리스트

현재까지 적용된 항목은 제외하고, 남은 작업만 정리하면 다음과 같다.

### 6-1. 캐시 무효화 범위 확장

지금은 삭제와 체크 상태 변경 중심으로 캐시 무효화를 보강했다.  
다음 단계에서는 작성, 수정, 제출 완료 후 복귀 동선까지 포함해서 캐시 정합성을 더 촘촘하게 맞출 필요가 있다.

- 작성 후 목록 복귀 시 어떤 캐시를 즉시 무효화할지 정리
- 수정 후 상세와 목록이 함께 갱신되도록 흐름 통일
- 제출 상태 변경 이후 관련 페이지 캐시 정합성 점검

### 6-2. 성능 측정 기준 정리

현재는 체감 검증과 수동 비교 중심으로 개선 효과를 확인하고 있다.  
다음에는 수치로 남길 수 있는 측정 기준도 보강하면 좋다.

- 첫 목록 진입 시간
- 페이지 재방문 시간
- 상세 첫 진입 시간
- 상세 재열기 시간

이 기준이 있으면 이후 리팩토링이나 추가 최적화 때 회귀 여부를 더 쉽게 확인할 수 있다.

### 6-3. 코드 리팩토링

지금 `ExpenseReport.jsx`에는 목록 조회, 페이지네이션, 캐시, 체크 상태 변경, 상세 열기, 상세 캐시, 삭제 로직이 함께 들어 있다.  
기능은 동작하지만, 유지보수성과 테스트 용이성 측면에서는 한 번 더 정리할 가치가 있다.

우선순위가 높은 리팩토링 항목은 다음과 같다.

- 목록 관련 캐시 로직을 별도 유틸 또는 커스텀 훅으로 분리
- 상세 관련 캐시 로직을 별도 유틸 또는 커스텀 훅으로 분리
- `useExpenseReportList`, `useExpenseReportDetail` 형태로 상태 책임 분리
- 로딩, 최신화, 에러, 무효화 상태 이름과 흐름을 일관된 패턴으로 정리
- 페이지 컴포넌트와 표시용 UI 컴포넌트의 책임 분리

### 6-4. 테스트 보강

캐시와 요청 경쟁 방어가 들어간 만큼, 수동 확인만으로는 놓치는 케이스가 생길 수 있다.

- 캐시 키 생성과 TTL 판정 테스트
- 캐시 무효화 테스트
- 페이지 재방문 시 즉시 표시되는 흐름 테스트
- 상세 재열기 시 cached 상세가 먼저 보이는 흐름 테스트
- 늦게 도착한 응답이 현재 화면을 덮어쓰지 않는지 테스트

## 7. 정리

이번 트러블슈팅은 “API를 빠르게 만든다”에서 끝나는 작업이 아니었다.  
권한 구조를 안전하게 유지하면서, 목록과 상세를 분리하고, 사용자가 실제로 느끼는 속도를 단계적으로 개선하는 작업에 가까웠다.

현재까지는 다음 기반을 확보했다.

- 목록과 상세를 각자 최적화할 수 있는 데이터 구조
- 권한 구조를 유지한 작성자명 표시 안정화
- 목록/상세 재방문 속도를 개선하는 캐시 전략
- 로딩 상태를 숨기지 않고 명확히 드러내는 UX

다음 단계는 이 구조를 더 단단하게 만드는 일이다.  
즉, 캐시 무효화 범위를 넓히고, 측정 기준을 세우고, 코드 구조를 리팩토링해 이후 유지보수 비용까지 함께 낮추는 방향으로 이어가면 된다.
