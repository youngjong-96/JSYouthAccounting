# 지출결의서 보기 페이지 로딩 속도 개선 트러블슈팅

## 1. 문제 정의

지출결의서 보기 페이지는 사용자 입장에서 가장 자주 접근하는 화면 중 하나다.  
특히 회계팀이나 마스터 계정은 목록을 여러 페이지 넘겨가며 확인하고, 각 문서의 상세 내용도 자주 열어보게 된다.

이번 개선 작업을 시작하게 된 계기는 크게 세 가지였다.

1. 작성자명 표시 문제를 안전하게 해결하는 과정에서 읽기 전용 API가 추가되었다.
2. 지출결의서 수가 늘어나면서 목록 조회 응답 속도와 화면 밀도가 모두 UX 이슈로 드러나기 시작했다.
3. 상세 보기를 눌렀을 때 로딩 중이라는 피드백이 거의 없어 사용자가 멈춘 것으로 오해할 수 있었다.

즉, 단순히 "API 응답 시간을 줄이는 것"만이 아니라 아래 세 가지를 함께 해결해야 했다.

- 첫 진입 시 목록이 더 빨리 보여야 한다.
- 다음 페이지로 이동할 때도 기다리는 느낌이 줄어야 한다.
- 상세 보기를 눌렀을 때 즉시 반응이 보여야 한다.

## 2. 개선 전 구조에서 보였던 한계

초기 구조에서는 지출결의서 목록과 상세 조회 최적화 포인트가 충분히 분리되어 있지 않았다.  
권한별로 보여줄 수 있는 데이터 범위와 작성자명 조합까지 고려해야 했기 때문에, 프론트엔드에서 단순히 화면만 그리는 방식으로는 해결이 어려웠다.

그 결과 아래와 같은 문제가 생겼다.

- 목록 화면 최적화와 상세 화면 최적화를 따로 가져가기 어려웠다.
- 작성자명, 권한 범위, 체크 상태 등 읽기 규칙이 프론트에서 일관되게 유지되기 어려웠다.
- 목록 조회와 상세 조회가 모두 느릴 때 어느 지점이 병목인지 구분하기 어려웠다.

## 3. 이번에 먼저 적용한 1차 개선

### 3-1. 서버 기준 페이지네이션 도입

지출결의서 보기 페이지는 PC와 모바일 모두 한 번에 5개씩만 보이도록 조정했다.  
이 과정에서 단순히 프론트에서 잘라 보여주는 방식이 아니라, 서버가 필요한 페이지 범위만 내려주도록 변경했다.

적용 이유는 다음과 같다.

- 처음부터 모든 문서를 한 번에 보내지 않도록 해서 응답 크기를 줄일 수 있다.
- 문서 수가 계속 늘어나도 목록 조회 비용이 비교적 안정적으로 유지된다.
- PC와 모바일에서 동일한 페이지 단위 UX를 유지할 수 있다.

### 3-2. 목록 API와 상세 API 분리

목록용 API와 상세용 API를 분리했다.

- 목록 API: 리스트 화면에 필요한 경량 정보만 반환
- 상세 API: 문서 본문, 항목, 영수증 등 상세 모달에 필요한 데이터 반환

적용 이유는 다음과 같다.

- 목록 화면에서 상세용 데이터를 불필요하게 함께 읽지 않아도 된다.
- 목록과 상세를 각각 다른 방식으로 최적화할 수 있다.
- 권한 구조를 유지하면서도 읽기 책임을 더 명확하게 나눌 수 있다.

### 3-3. 작성자명 조회를 서버에서 안전하게 조합

마스터 계정에서는 작성자명이 보이는데 회계팀 계정에서는 `작성자 미상`처럼 보이던 문제를 해결하기 위해, 작성자명 조합 로직을 서버 읽기 API에 포함시켰다.

적용 이유는 다음과 같다.

- 권한 구조를 깨지 않고 일관된 작성자명 표시가 가능하다.
- 프론트가 역할별 예외 처리를 과도하게 갖지 않아도 된다.
- 목록과 상세 모두 같은 기준으로 작성자명을 표시할 수 있다.

## 4. 현재까지의 결과

배포 환경에서 확인했을 때, 체감 기준으로 이전보다 약 2초 정도 빨라졌다.  
특히 첫 목록 진입과 작성자명 표시 안정성 측면에서는 의미 있는 개선이 있었다.

현재까지 확인된 성과는 다음과 같다.

- 회계팀 계정에서도 작성자명이 정상적으로 표시된다.
- 목록 조회가 서버 페이지네이션 기반으로 바뀌어 응답 부담이 줄었다.
- 상세 조회 로직이 별도 API로 분리되어 이후 최적화 기반이 마련되었다.

다만 아직 남은 문제가 있다.

- 페이지를 넘길 때마다 API 응답을 기다려야 해서 UX가 여전히 끊긴다.
- 상세 보기를 눌렀을 때 로딩 중이라는 시각적 피드백이 부족하다.
- 이미 본 페이지나 이미 연 상세 문서를 다시 볼 때도 캐시 이점이 거의 없다.

## 5. 남은 개선 계획

현재까지 구현된 범위는 유지하고, 아래 항목을 다음 단계로 진행한다.

### 5-1. 목록 캐시 추가

페이지 번호, 필터값, 권한 기준을 묶어 캐시 키를 만들고 목록 데이터를 메모리에 캐시한다.

- 한 번 본 페이지는 즉시 다시 보여준다.
- 뒤에서는 최신 데이터를 다시 확인하는 `stale-while-revalidate` 방식으로 갱신한다.

### 5-2. 페이지 이동 UX 개선

다음 페이지나 이전 페이지를 누를 때 기존 목록을 먼저 비우지 않고 유지한다.  
대신 목록 영역이나 페이지 버튼 근처에 로딩 상태를 명확하게 표시한다.

- 버튼 비활성화
- 현재 페이지 로딩 스피너 표시
- 목록 상단 또는 표 영역의 "불러오는 중" 안내

그리고 현재 페이지 로딩이 끝난 뒤에는 다음 페이지를 백그라운드에서 미리 가져오는 방식도 적용한다.

### 5-3. 상세 보기 로딩 피드백 개선

상세 보기 버튼을 눌렀을 때 즉시 반응이 보이도록 UX를 바꾼다.

- 상세 모달을 먼저 열고 내부는 스켈레톤 또는 로딩 안내를 표시
- 클릭한 상세 보기 버튼에는 작은 스피너 표시
- 중복 클릭 방지

이 단계의 핵심은 "실제로 빠르지 않더라도 멈춘 것처럼 보이지 않게 만드는 것"이다.

### 5-4. 상세 데이터 캐시 추가

`reportId` 기준으로 상세 데이터를 캐시한다.

- 한 번 연 상세 문서는 다시 열 때 즉시 표시
- 필요하면 뒤에서 최신 데이터 재검증

### 5-5. 요청 경합 방지

페이지를 빠르게 넘기거나 상세 보기를 연속 클릭할 때 이전 요청이 늦게 도착해 화면을 덮어쓰지 않도록 정리한다.

- 마지막 요청만 유효하게 처리
- 오래된 응답 무시
- 필요 시 요청 취소 또는 요청 식별자 비교 적용

### 5-6. 캐시 무효화 규칙 정리

작성, 수정, 삭제, 체크 상태 변경 이후에는 관련 목록/상세 캐시를 정확히 무효화한다.

- 현재 페이지 목록 갱신
- 해당 상세 캐시 제거 또는 최신화
- 필터별 캐시 오염 방지

## 6. 이 방법을 선택한 이유

이번 개선은 단순히 "응답 시간을 줄이는 코드"보다 "사용자가 느끼는 속도"를 우선으로 잡았다.

그 이유는 다음과 같다.

- 실제 업무 화면에서는 첫 진입만 빠른 것보다 페이지 이동과 상세 전환이 부드러운 것이 더 중요하다.
- 서버 페이징만으로는 네트워크 왕복 자체가 사라지지 않기 때문에 캐시와 선로딩이 함께 필요하다.
- 권한 구조와 작성자명 조합 로직은 서버에 두는 편이 안전하므로, 프론트는 표시와 체감 속도 개선에 집중하는 편이 맞다.

## 7. 이번 범위에서 제외한 방법

이번 계획에서는 `PC hover prefetch`는 제외했다.

제외 이유는 다음과 같다.

- PC 환경에서만 동작하는 전용 UX라서 모바일 공통 개선 효과가 없다.
- 사용자가 실제로 열지 않을 상세를 과도하게 미리 읽을 가능성이 있다.
- 현재 단계에서는 캐시, 페이지 전환 UX, 상세 로딩 피드백이 우선순위가 더 높다.

즉, 지금은 "호버 시점의 추가 최적화"보다 "모든 기기에서 공통으로 체감되는 개선"을 먼저 적용하는 것이 맞다고 판단했다.

## 8. 구현 진행 메모

목록 캐시와 페이지 이동 UX 개선 작업까지 진행한 현재 시점에는 아래 항목이 추가로 적용된 상태다.

- 페이지 번호, 필터, 사용자 기준으로 목록 페이지 캐시를 분리해서 같은 페이지 재방문 시 즉시 목록을 보여주도록 변경
- 캐시된 목록을 먼저 보여주고 뒤에서 다시 최신 데이터를 확인하는 `stale-while-revalidate` 방식 적용
- 현재 페이지를 읽은 뒤 다음 페이지를 백그라운드에서 선로딩하도록 구성
- 페이지 이동 시 기존 목록을 비우지 않고 유지한 채, 상단 배지/오버레이/페이지 버튼 스피너로 로딩 상태를 명확하게 표시
- 삭제와 체크 상태 변경 이후에는 관련 목록 캐시를 비우고 현재 페이지를 다시 검증하도록 보강

아직 남은 핵심 작업은 상세 보기 로딩 피드백과 상세 캐시다.  
즉, 이제부터는 "목록 이동 체감 속도"를 다듬는 단계에서 "상세 열기 체감 속도"를 다듬는 단계로 넘어간다고 볼 수 있다.

## 9. 중간 결론

이번 작업은 단번에 끝나는 성능 개선이라기보다, 구조 정리와 UX 최적화를 단계적으로 쌓아가는 과정에 가깝다.

현재까지는 아래 두 가지를 확보했다.

- 목록과 상세를 분리해서 각자 최적화할 수 있는 기반
- 권한 구조를 유지하면서도 작성자명과 목록 표시를 안정적으로 처리할 수 있는 읽기 API

다음 단계에서는 캐시와 로딩 피드백을 중심으로, "실제 응답 시간"과 "체감 속도"를 함께 개선하는 방향으로 이어간다.
